# sr-obsidian:adr — 기술 의사결정 기록

대화에서 오간 기술적 고민을 ADR 형식으로 vault에 남긴다.
나중에 "왜 이걸 선택했지?"라는 질문에 답할 수 있게 하는 것이 목적이다.

## 사용법

```
/sr-obsidian:adr                          # 현재 대화에서 고민 자동 추출
/sr-obsidian:adr payment                  # 프로젝트 명시
/sr-obsidian:adr "NicePay SDK 버전 선택"  # 제목 명시
```

---

## Phase 1: 고민 추출 (READ-ONLY)

`$ARGUMENTS`가 있으면 힌트로 사용하고, 없으면 현재 대화에서 아래를 추출한다.

1. **문제 상황** — 왜 결정이 필요했는가
2. **검토한 선택지** — 고민했던 옵션들 (2개 이상)
3. **결정** — 무엇을 선택했는가
4. **이유** — 선택의 근거 (기술적 제약, 성능, 유지보수 등)
5. **트레이드오프** — 선택하지 않은 옵션의 장점 / 선택한 옵션의 단점
6. **프로젝트** — `20-areas/` 하위 어느 폴더에 넣을지

추출 후 아래 형식으로 출력하여 사용자 확인을 받는다:

---

### 📋 ADR 초안 — {제목}

| 항목 | 내용 |
|------|------|
| 프로젝트 | {프로젝트} |
| 제목 | {제목} |
| 결정 | {결정 한 줄 요약} |

**저장하시겠습니까?** (y / 수정사항 입력)

## Phase 2: 저장 (WRITE, 확인 후)

### 파일 경로

```
/Users/sr/obsidian/sr-labs/20-areas/{프로젝트}/adr/{YYYY-MM-DD}-{slug}.md
```

- `{slug}`: 제목을 소문자 kebab-case로 변환 (한글 포함 가능)
- `adr/` 폴더가 없으면 자동 생성

### 파일 형식

```markdown
---
type: permanent
adr: true
status: accepted
created: {YYYY-MM-DD}
project: {프로젝트}
tags: [adr, {프로젝트}]
---

# ADR: {제목}

**상태**: Accepted  
**날짜**: {YYYY-MM-DD}

## 문제 상황

{왜 이 결정이 필요했는가. 기술적 배경, 제약사항, 발견된 문제를 서술한다.}

## 검토한 선택지

### 선택지 1: {이름}

{설명}

- 장점: {장점}
- 단점: {단점}

### 선택지 2: {이름}

{설명}

- 장점: {장점}
- 단점: {단점}

(선택지가 더 있으면 추가)

## 결정

**{선택한 것}**을 선택한다.

{결정 이유 — 기술적 제약, 팀 합의, 비용, 유지보수 관점 등을 구체적으로 서술한다.}

## 트레이드오프

- 포기한 것: {선택하지 않은 옵션의 장점}
- 감수하는 것: {선택한 옵션의 단점 또는 위험}

## 관련 자료

- {파일 경로, URL, 참고 문서 등}
```

### 저장 후 처리

1. `60-logs/ingest-log.md` 말미에 append:
   ```
   ## [YYYY-MM-DD] adr | {제목}
   - 20-areas/{프로젝트}/adr/{파일명}.md
   - 결정: {결정 한 줄 요약}
   ```

2. 커밋 — commit-msg 훅이 `#숫자` 이슈 번호를 필수로 검사한다.
   사용자에게 이슈 번호를 묻고, 받은 번호로 커밋한다:
   ```bash
   cd /Users/sr/obsidian/sr-labs
   git add 20-areas/{프로젝트}/adr/{파일명}.md 60-logs/ingest-log.md
   git commit -m "docs: ADR {제목} (#{이슈번호})"
   ```
   이슈 번호를 모르거나 없는 경우: `gh issue create --repo SeokRae/knowledge-labs --title "ADR: {제목}"` 실행 후 반환된 번호 사용.

3. 완료 출력:
   ```
   ✅ ADR 저장 완료
   파일: 20-areas/{프로젝트}/adr/{파일명}.md
   ```

## 판단 기준

| 상황 | 처리 |
|------|------|
| 선택지가 1개뿐 | "고민이 없었다면 ADR보다 capture가 적합합니다" 안내 |
| 결정이 아직 미확정 | status: proposed 로 저장, 나중에 Edit으로 갱신 |
| 같은 주제 ADR이 이미 존재 | 기존 파일 Read 후 "갱신"할지 "신규"로 남길지 확인 |
| 프로젝트 불명확 | "어느 프로젝트의 결정인가요?" 질문 |
